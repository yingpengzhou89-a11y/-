# 竞技场任务逻辑审查与修改建议

## 1. 审查对象

当前审查函数：

```python
def run_arena_task_logic(self, observation):
    """独立包装：竞技场挑战各阶段控制与退出逻辑"""
```

该函数的作用是根据当前页面 OCR 文本 `page_text`，判断竞技场所处阶段，并返回一次操作决策，例如：

- 点击某个坐标；
- 等待战斗结束；
- 关闭战斗结算；
- 关闭挑战列表；
- 停止任务并等待人工确认。

它本质上是一个“基于 OCR 文本的竞技场状态决策器”，但目前还不是一个完整、可靠的竞技场任务状态机。

---

## 2. 目标任务流程

完整任务应当按以下流程执行：

1. 在任务栏中找到竞技场任务并点击“前往”；
2. 跳转到竞技场主页；
3. 点击右下角“挑战”按钮；
4. 进入挑战列表；
5. 判断“自动战斗”是否已经勾选：
   - 已经勾选：不要再次点击；
   - 未勾选：尝试勾选；
6. 如果出现“进行3次战斗后方可跳过”之类提示：
   - 关闭提示；
   - 不再反复点击自动战斗；
   - 正常观看本场战斗；
7. 点击挑战列表中的“免费”按钮；
8. 战斗结束后关闭结算页面；
9. 返回挑战列表，读取“今日次数：x/50”；
10. 如果当前挑战次数仍未达到任务要求，继续挑战；
11. 达到任务要求后关闭挑战列表；
12. 返回任务栏领取任务奖励。

补充规则：

- 自动战斗需要本周累计完成至少3次战斗后才能启用；
- 自动战斗没有解锁时，必须正常等待整场战斗完成；
- 完成第3次后，应允许程序再次尝试勾选自动战斗；
- 一键战斗需要累计10次后解锁，但本阶段暂不实现；
- 只点击显示为“免费”的挑战按钮，避免消耗竞技券；
- 实际成功挑战次数必须以“今日次数：x/50”为准，不能仅依据点击次数推断。

---

## 3. 总体结论

当前代码只能覆盖竞技场流程中的一部分，无法稳定完成完整任务。

主要缺陷包括：

1. 缺少竞技场主页右下角“挑战”按钮的点击逻辑；
2. 没有真正判断自动战斗是否已经勾选；
3. 把“是否曾经点击过自动战斗”错误地当成“当前是否已勾选”；
4. 没有处理“进行3次战斗后方可跳过”的未解锁提示；
5. 自动战斗未解锁时，代码会点击“挂起”，与正常观看战斗的要求冲突；
6. 包含当前明确不需要的一键战斗逻辑；
7. 使用点击决策次数代替实际成功挑战次数；
8. 将任务目标次数与免费挑战上限混用；
9. OCR 关键词中存在过于宽泛的乱码字符，容易误判；
10. 达成任务后只关闭挑战列表，没有返回任务栏领取奖励；
11. 某些地方遇到一次 OCR 漏识别就直接停止，容错性不足。

## 4. 坐标问题的修正说明

用户实际模拟器分辨率约为：

```text
1280 × 720
```

上传的图片文件虽然是：

```text
1920 × 1080
```

但不能直接根据上传图片的像素位置，断定代码中的点击坐标错误。

代码中的坐标可能属于以下任一种情况：

- 模拟器原始坐标；
- ADB 截图坐标；
- 程序内部基准分辨率坐标；
- 经过缩放映射后的逻辑坐标。

因此，本文不再把现有坐标认定为“确定错误”，而是标记为：

> 需要确认点击执行层使用的实际坐标系，并通过运行日志或点击可视化验证。

建议检查：

```python
observation["screen_width"]
observation["screen_height"]
```

以及点击执行代码是否有类似缩放：

```python
actual_x = base_x * current_width / base_width
actual_y = base_y * current_height / base_height
```

如果当前坐标已经在1280×720模拟器中经过实际测试，则可以继续保留。

识别到战斗胜利、失败或继续提示后，点击继续坐标。

- `"WIN"` 可能在其他英文文本中误匹配；

2. `"确定"` 重复写了两次；
3. OCR 可能把其他页面中的“一键战斗”识别出来，造成误点；
4. 当前目标是单次免费挑战，不应主动进入批量战斗流程；
5. 固定确认坐标未放入配置，不利于统一管理。

建议：

```python
# 暂时删除或关闭一键战斗相关逻辑
```

### 5.4 战斗内点击“挂起”

```python
if text_contains_any(page_text, arena_config.get("battle_keywords") or []):
    if not self.has_decision("点击挂起跳过战斗"):
        return {
            "intent": "点击挂起跳过战斗",
            "action": "tap",
            "target": {"x": 45, "y": 50}
        }
```

这是当前最明显的业务逻辑冲突之一。

任务规则是：

- 自动战斗已解锁并勾选时，游戏自己跳过战斗过程；
- 自动战斗未解锁时，应正常观看整场战斗。

现有逻辑却会：

- 一进入战斗就点击“挂起”；
- 把“挂起”当成跳过战斗的方法。

风险：

- “挂起”未必等于“跳过”；
- 可能打开暂停、托管或其他界面；
- 可能阻塞战斗；
- 与任务要求不一致。

建议替换为：

```python
if text_contains_any(page_text, arena_config.get("battle_keywords") or []):
    return {
        "intent": "等待竞技场战斗结束",
        "action": "wait",
        "target": {"seconds": 8},
        "confidence": 0.86,
        "risk": "low"
    }
```

等待后重新截图判断，直到进入结算页。

### 5.5 挑战列表识别

```python
if text_contains_any(
    page_text,
    (arena_config.get("challenge_list_keywords") or [])
    + ["/50", "սб", "Զս", "ˢ"]
):
```

问题：

- `"/50"` 可以作为“今日次数”区域的辅助特征；
- `"սб"`、`"Զս"`、`"ˢ"` 等乱码字符过于宽泛；
- 特别是单字符 `"ˢ"` 很容易在无关页面被识别到；
- 可能导致程序把其他页面误判为挑战列表。

建议使用更稳定的组合条件，例如：

```python
is_challenge_list = (
    text_contains_any(page_text, ["挑战列表"])
    or (
        text_contains_any(page_text, ["今日次数", "/50"])
        and text_contains_any(page_text, ["免费", "自动战斗"])
    )
)
```

如果 OCR 经常漏字，建议为以下区域单独做 ROI OCR：

- 标题“挑战列表”；
- “今日次数：x/50”；
- 自动战斗文字；
- 第一排免费按钮。

## 6. 自动战斗逻辑的核心错误

现有代码：

```python
if not self.has_decision("勾选挑战列表自动战斗"):
    return {
        "intent": "勾选挑战列表自动战斗",
        "action": "tap",
        "target": {"x": 603, "y": 202}
    }
```

该判断实际表达的是：

> 本轮任务执行历史中，是否已经发出过一次“勾选自动战斗”的点击决策。

但任务真正需要判断的是：

> 当前页面上的自动战斗勾选框，此刻到底处于已勾选还是未勾选状态。

这两者完全不同。

### 可能出现的错误情况

#### 情况1：自动战斗原本已经勾选

程序看不到真实勾选状态，只发现历史里没点过，于是再次点击。

结果：

```text
已勾选 → 被点击 → 取消勾选
```

#### 情况2：程序发出了点击，但点击没有生效

历史中已经记录该决策，后续不会重试。

结果：

```text
自动战斗仍未勾选，但程序以为已经处理过
```

#### 情况3：未达到3次，点击后出现未解锁提示

程序会认为自己已经尝试完成，后续不再点击。

当挑战次数达到3次以后，也不会重新尝试启用自动战斗。

#### 情况4：任务重新进入挑战列表

如果 `has_decision` 的历史没有在合适时机重置，可能导致后续任务永远不再检查自动战斗。

## 7. 自动战斗的正确实现方式

应在视觉识别层中增加一个状态字段：

```python
observation["arena_auto_battle_checked"]
```

推荐值：

```python
True   # 已勾选
False  # 未勾选
None   # 无法确定
```

判断方法可以是：

1. 裁剪自动战斗勾选框 ROI；
2. 模板匹配“已勾选”和“未勾选”两种状态；
3. 颜色、亮度或图形特征判断；
4. 必要时结合点击前后两帧差异。

示例：

```python
auto_checked = observation.get("arena_auto_battle_checked")

if auto_checked is True:
    # 已经勾选，不操作
    pass

elif auto_checked is False:
    # 当前次数下尚未尝试过，才点击
    if self.arena_last_auto_try_count != today_count:
        self.arena_last_auto_try_count = today_count
        return {
            "intent": "尝试勾选竞技场自动战斗",
            "action": "tap",
            "target": arena_config["auto_battle_point"],
            "confidence": 0.88,
            "risk": "low"
        }

else:
    # 无法判断时不要盲点，可等待重识别
    return {
        "intent": "自动战斗状态无法确认，等待重新识别",
        "action": "wait",
        "target": {"seconds": 1},
        "confidence": 0.7,
        "risk": "low"
    }
```

## 8. 为什么要记录“上次尝试时的今日次数”

自动战斗在累计3次战斗后解锁。

假设当前：

```text
今日次数：2/50
```

程序点击自动战斗，弹出：

```text
进行3次战斗后方可跳过
```

此时不应在同一个 `2/50` 状态下无限重复点击。

可以记录：

```python
self.arena_last_auto_try_count = 2
```

完成一场后变为：

```text
今日次数：3/50
```

此时：

```python
self.arena_last_auto_try_count != today_count
```

程序就可以再次尝试勾选。

这样能够同时实现：

- 未解锁时避免无限点击；
- 挑战次数变化后自动重试；
- 达到3次后顺利开启自动战斗。

## 9. 缺少未解锁提示处理

当前代码没有处理：

```text
进行3次战斗后方可跳过
```

或类似文本。

应当在挑战列表普通逻辑之前增加高优先级分支：

```python
if text_contains_any(page_text, [
    "进行3次战斗后方可跳过",
    "3次战斗后方可跳过",
    "3次战斗后",
    "方可跳过"
]):
    return {
        "intent": "关闭自动战斗未解锁提示",
        "action": "tap",
        "target": arena_config["auto_battle_locked_confirm_point"],
        "confidence": 0.9,
        "risk": "low"
    }
```

为什么必须高优先级：

- 弹窗出现时，OCR 仍可能识别到背景中的“免费”“今日次数”“自动战斗”；
- 如果先进入挑战列表逻辑，程序可能错误点击背景按钮；
- 弹窗应优先关闭，再继续后续操作。

## 10. 缺少竞技场主页“挑战”按钮逻辑

完整流程中，从任务栏“前往”后会进入竞技场主页。

该页面右下角存在“挑战”按钮。

当前函数中没有一个明确分支用于：

```text
竞技场主页 → 点击右下角挑战 → 打开挑战列表
```

因此程序可能停在竞技场主页，函数最终返回：

```python
return None
```

应增加类似分支：

```python
if (
    text_contains_any(page_text, ["竞技场", "今日免费"])
    and text_contains_any(page_text, ["挑战"])
    and not text_contains_any(page_text, ["挑战列表", "/50"])
):
    return {
        "intent": "打开竞技场挑战列表",
        "action": "tap",
        "target": arena_config["challenge_button_point"],
        "confidence": 0.88,
        "risk": "low"
    }
```

注意：

- 不能只凭“挑战”两个字判断，因为挑战列表里也有挑战按钮；
- 最好结合竞技场主页特征；
- 也可以增加右下角按钮模板匹配。

## 11. 实际挑战次数判断错误

现有代码：

```python
free_challenges = self.decision_count(
    intent_contains="竞技场免费挑战"
)
```

它统计的是：

```text
程序发出了多少次“点击免费挑战”的决策
```

它并不等于：

```text
成功完成了多少次竞技场挑战
```

点击可能失败，例如：

- 按钮被弹窗遮挡；
- 坐标偏移；
- 网络延迟；
- 点击后页面没有响应；
- OCR 误判；
- 点击到了错误区域。

因此不能使用：

```python
free_challenges
```

作为任务完成进度。

实际进度必须来自：

```text
今日次数：x/50
```

也就是：

```python
today_count = arena_today_count(page_text)
```

推荐逻辑：

```python
if today_count is None:
    return 等待重新识别

if today_count >= target_today_count:
    return 关闭挑战列表

if 页面存在免费按钮:
    return 点击免费按钮
```

点击后等待页面变化，并通过下一次 OCR 中 `today_count` 增加来确认成功。

## 12. `max_free_challenges` 的语义混乱

现有代码：

```python
max_free_challenges = int(
    arena_config.get("max_free_challenges", 3)
)
```

随后又用它判断：

```python
if today_count >= max_free_challenges:
```

这里把两个概念混在了一起：

### 概念1：任务需要完成的挑战次数

例如：

```python
target_today_count = 3
```

### 概念2：当天可用的免费挑战次数

例如游戏页面显示：

```text
今日免费4次
```

建议拆分配置：

```python
target_today_count = int(
    arena_config.get("target_today_count", 3)
)

max_free_attempts = int(
    arena_config.get("max_free_attempts", 4)
)
```

实际任务完成判断只使用：

```python
today_count >= target_today_count
```

而是否消耗竞技券则通过按钮是否显示“免费”来保护，不应只依赖固定上限。

```python
if (
```

2. 一键战斗有额外解锁条件；

## 14. 免费按钮判断仍需加强

现有代码：

```python
if free_challenges < max_free_challenges \
        and text_contains_any(page_text, ["免费", "ť"]):
```

问题：

- `"ť"` 是极不稳定的单字符乱码；
- 整页任意位置出现“免费”都可能触发；
- 页面上可能有“今日免费4次”，但挑战按钮本身已经不是免费；
- 只检测到文字，不代表固定坐标处的按钮就是免费按钮。

推荐方法：

1. 对第一排挑战按钮区域做 ROI OCR；
2. 确认该区域识别为“免费”；
3. 再点击对应按钮；
4. 若第一排不是免费，可以扫描第二、第三排；
5. 若所有可见按钮都不是免费，立即停止，避免消耗竞技券。

建议 observation 中增加：

```python
observation["arena_free_button_points"]
```

例如：

```python
[
    {"x": 995, "y": 235},
    {"x": 995, "y": 335}
]
```

具体坐标以实际1280×729坐标系为准。

## 15. 达到次数后缺少领取奖励流程

当前代码达到次数后只执行：

```python
关闭挑战列表
```

但完整任务还需要：

```text
返回任务栏 → 找到任务 → 点击领取
```

这部分可以放在：

- 竞技场函数后续状态中；
- 或外层任务控制器中。

推荐在关闭挑战列表时返回一个明确完成标记：

```python
return {
    "intent": "竞技场挑战次数已满足，关闭挑战列表",
    "action": "tap",
    "target": arena_config["challenge_list_close_point"],
```

## 16. OCR 漏识别时不应立即停止

当前末尾逻辑：

```python
return {

    ...

```

风险：

- 一次 OCR 没识别出“免费”就停止；
- 页面仍在加载；
- 按钮动画导致文字模糊；
- 弹窗刚关闭，画面尚未稳定；
- OCR 短暂错误。

更稳妥的方式：

```python
if self.arena_no_action_retry_count < 3:

        "intent": "竞技场页面识别不完整，等待重试",
        "action": "wait",
        "target": {"seconds": 1},
        "confidence": 0.65,
        "risk": "low"
    }

return {
    "intent": "连续多次无法确认免费挑战入口，停止等待人工确认",
    "action": "stop",
    ...
}
```

只在连续多次识别失败后再停止。
