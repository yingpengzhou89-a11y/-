# 日常任务自动化交接文档

生成时间：2026-06-15

## 项目目标

在现有 `resource_detector` 项目上扩展一个 MuMu 模拟器 + ADB 的日常任务自动化 MVP。当前目标不是做全自动通用游戏 AI，而是先把常见日常任务做成可观察、可回放、带护栏的半规则自动化流程。

核心循环：

```text
截图 observe -> OCR/规则 understand -> 决策 plan -> ADB 点击/滑动/等待 act -> 再截图 verify
```

每次运行都会生成 `task_runs/<run_id>/`，里面包含步骤截图和 `events.jsonl`，方便回放调试。

## 技术栈

- Python
- OpenCV：截图缩放、图像 hash、基础图像处理
- RapidOCR：识别游戏界面中文字
- ADB：控制 MuMu 模拟器截图、点击、滑动
- MuMu 模拟器：当前测试设备为 `127.0.0.1:7555`
- JSON 配置：坐标、关键词、护栏、任务规则都放在 `config.json`
- 可选云端视觉模型接口：`task_ai.py` 已有 OpenAI-compatible 风格 HTTP 客户端，但当前实际流程主要靠规则和 OCR，AI 没启用

## 关键文件

- `daily_tasks.py`
  - 日常任务入口脚本
  - 支持 `--image` 离线分析截图
  - 支持 `--dry-run` 只输出下一步决策，不真实点击
  - 支持 `--target` 指定目标任务，例如 `友情点`、`招募`、`竞技`

- `task_detector.py`
  - OCR 识别任务页
  - 解析任务标题、进度、按钮状态
  - 可识别 `领取`、`前往`、`已领取`
  - 针对 OCR 漏读绿色 `已领取` 的情况，增加了“任务标题要求次数 + 进度左值已达标”推断逻辑

- `task_runner.py`
  - 主自动化循环
  - 包含任务规则、动作执行、护栏、日志
  - 当前已包含友情点、招募、竞技场的部分规则

- `task_ai.py`
  - 云端视觉模型接口
  - 当前没有启用
  - 如果后续启用，建议接阿里云百炼的 OpenAI-compatible endpoint，并使用快且便宜的 Qwen 多模态模型

- `actions.py`
  - ADB 点击、滑动、等待封装
  - 文件里仍有 `adb_back`，但日常任务流程中不要使用它

- `adb_client.py`
  - ADB 截图和自动连接 MuMu TCP 设备
  - 当前运行配置在 `config_runtime_7555.json`

- `config.json`
  - 主配置文件
  - 新增了 `task_page`、`recruitment`、`arena`、`guardrails`、`known_tasks`

## 重要护栏和注意事项

### 不要使用 ESC / Android Back

用户明确说过：

- 游戏界面按 ESC 可能会直接关闭游戏
- 不要模拟 ESC
- 不要用 Android 返回键

所以日常任务自动化里只允许：

- tap
- swipe
- wait
- claim
- stop

### 不要处理 MuMu 桌面回游戏

用户明确说过：

- 不需要从 MuMu 桌面重新进游戏
- 因为登录、公告、活动弹窗太多
- 自动化起点先限制为游戏主界面或任务页

### 禁止高风险点击

全局禁词包括：

- `充值`
- `购买`
- `钻石`
- `付费`
- `商城`
- `确认购买`
- `代币`
- `点券`
- `不足`

注意：有些正常界面会带 `商城` 或 `购买` 字样，所以 `task_runner.py` 里有少量白名单 intent，例如关闭弹窗、进入竞技场、免费挑战等。新增规则时要谨慎加白名单。

## 已知问题

### 1. `is_task_page` 对非任务页偶尔误判

好友页、竞技页等页面会被 OCR 解析出一些伪 task row，这是正常现象。真正判断任务页主要靠 `日常任务` 关键词。

### 2. OCR 会把字识别错

常见错误：

- `挑战` 被读成 `桃战`
- `一键战斗` 被读成 `键战斗`
- 绿色 `已领取` 经常漏读
- `30/5` 后面可能粘上奖励数量，变成 `30/5100`

所以规则里不要依赖单一文本，最好用多个关键词和进度兜底。

### 3. 竞技场战斗时间很长

一场战斗可能要 1 分钟以上。当前规则是战斗中每次 `wait 8 seconds`，靠 OCR 识别 `挂起/自动/x2/暂停` 判断还在战斗。

这会消耗较多 OCR 时间，但相对安全。

### 4. 竞技场挑战列表含有 `自动战斗`

之前 bug：

- 挑战列表也有 `自动战斗`
- 导致被误判为战斗中等待

已修：

- 挑战列表规则优先于战斗中等待规则

## 模型/API 建议

用户不想用 OpenAI API key，已有：

- DeepSeek
- 阿里云百炼
- 火山方舟
- 硅基流动

建议优先试阿里云百炼的 Qwen 多模态模型，原因：

- OpenAI-compatible 接口容易接入现有 `task_ai.py`
- 中文 OCR/游戏 UI 理解相对友好
- 这类任务对深度推理要求不高，更看重速度、价格、视觉识别稳定性

但当前 MVP 不建议一开始就让云端模型高自主点击。更稳妥的模式是：

```text
规则优先 -> 模型只做页面解释或坐标建议 -> 动作必须经过白名单和禁词护栏
```
