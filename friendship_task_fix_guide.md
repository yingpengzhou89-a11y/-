# 友情点赠送任务低概率死循环修复指南

## 1. 文档目的

请修改友情点赠送任务代码，解决下面这种低概率死循环：

```text
进入好友列表
→ 没有点击右下角“领取赠送”
→ 直接点击左上角小房子返回主界面
→ 从任务列表再次点击友情点任务“前往”
→ 再次进入好友列表
→ 仍然跳过“领取赠送”
→ 再次返回主界面
→ 无限重复
```

本次修改应以“最小风险修复”为主，不要改动任务入口、页面分类器、任务调度顺序以及现有坐标体系。

---

## 2. 当前代码中的核心问题

当前代码使用下面的判断确认领取赠送动作是否已经完成：

```python
used_friendship_action = runner.has_decision(
    "友情点页面执行领取赠送"
)
```

但 `runner.has_decision()` 最多只能证明：

> 历史决策中曾经生成过“友情点页面执行领取赠送”这个 intent。

它不能证明：

- 点击真正执行成功；
- 点击落在了正确位置；
- 游戏已经收到点击；
- 页面当时没有加载动画或遮挡；
- 网络请求已经完成；
- “领取和赠送成功”已经出现；
- 这条历史记录属于当前这一轮友情点任务。

因此，当前代码实际上把：

```text
曾经生成过点击决策
```

错误地等价成了：

```text
领取赠送已经成功
```

这会产生一个典型的小概率竞态问题。

---

## 3. 可能的故障过程

低概率故障可能按照下面的顺序发生：

1. 进入好友列表页面。
2. `run_task()` 返回“点击领取赠送”。
3. Runner先把这条 intent 写入历史决策。
4. 实际点击因页面加载、ADB延迟、动画遮挡或偶发丢失而没有生效。
5. 下一帧再次调用 `run_task()`。
6. `runner.has_decision("友情点页面执行领取赠送")` 返回 `True`。
7. 程序误认为领取赠送已经完成。
8. 直接点击小房子返回主界面。
9. 任务列表发现友情点任务仍未完成，再次进入好友页。
10. 历史决策仍然存在，所以程序继续跳过领取按钮。
11. 最终形成往返死循环。

这个问题属于“动作决策已记录，但动作结果未确认”。

---

## 4. 修改目标

修改后必须满足以下规则：

1. **没有检测到成功证据时，绝不能直接返回主界面。**
2. 第一次点击“领取赠送”后必须等待页面响应。
3. 等待后必须检查成功文本或完成状态。
4. 第一次没有确认成功时，允许再重试一次。
5. 第二次仍然无法确认时，应停止任务，不能继续回主页重进。
6. OCR文本必须先做格式归一化，避免冒号、空格、换行造成误判。
7. 保留现有坐标：
   - 领取赠送：`(1156, 653)`
   - 小房子返回：`(324, 39)`
8. 保持函数入口不变：

```python
def run_task(runner, observation):
```

## 5. 建议直接替换的代码

请将当前友情点任务文件中的 `run_task()` 替换为下面的实现。

```python
from task_detector import text_contains_any


def run_task(runner, observation):
    """友情点任务：执行领取赠送，确认成功后再退出。"""

    page_text = observation.get("page_text") or ""

    # OCR文本归一化：
    # 避免中文冒号、英文冒号、空格和换行导致完成状态匹配失败。
    normalized_text = (
        page_text
        .replace(" ", "")
        .replace("\n", "")
        .replace("\r", "")
        .replace("：", "")
        .replace(":", "")
    )

    # 1. 只有出现明确成功证据时，才允许返回主界面。
    #
    # “领取和赠送成功”表示本次点击已经生效；
    # “每日赠送上限30/30”表示今日赠送次数已满。
    friendship_success = (
        "领取和赠送成功" in normalized_text
        or "每日赠送上限30/30" in normalized_text
    )

    if friendship_success:
        return {
            "intent": "友情点赠送确认成功，回到主界面",
            "action": "tap",
            "target": {
                "x": 324,
                "y": 39
            },
            "confidence": 0.95,
            "risk": "low"
        }

    # 2. 当前运行历史中还没有发出过首次点击时，点击领取赠送。
    clicked_once = runner.has_decision(
        "友情点首次点击领取赠送"
    )

    if not clicked_once:
        return {
            "intent": "友情点首次点击领取赠送",
            "action": "tap",
            "target": {
                "x": 1156,
                "y": 653
            },
            "confidence": 0.88,
            "risk": "low"
        }

    # 3. 首次点击后必须等待游戏处理，不能下一帧立刻退出。
    waited_once = runner.has_decision(
        "友情点首次点击后等待确认"
    )

    if not waited_once:
        return {
            "intent": "友情点首次点击后等待确认",
            "action": "wait",
            "target": {
                "seconds": 2
            },
            "confidence": 0.95,
            "risk": "low"
        }

    # 4. 等待后仍然没有识别到成功证据，允许重试一次。
    retried_once = runner.has_decision(
        "友情点重试点击领取赠送"
    )

    if not retried_once:
        return {
            "intent": "友情点重试点击领取赠送",
            "action": "tap",
            "target": {
                "x": 1156,
                "y": 653
            },
            "confidence": 0.82,
            "risk": "low"
        }

    # 5. 重试后再次等待页面响应。
    waited_after_retry = runner.has_decision(
        "友情点重试后等待确认"
    )

    if not waited_after_retry:
        return {
            "intent": "友情点重试后等待确认",
            "action": "wait",
            "target": {
                "seconds": 2
            },
            "confidence": 0.95,
            "risk": "low"
        }

    # 6. 两次点击后仍然没有确认成功，安全停止。
    #
    # 这里绝不能点击小房子，否则任务列表仍会认为友情点任务未完成，
    # 从而重新进入好友列表并形成死循环。
    return {
        "intent": "友情点领取结果未确认，停止避免往返死循环",
        "action": "stop",
        "target": {
            "reason": "friendship action result not confirmed after retry"
        },
        "confidence": 1.0,
        "risk": "low"
    }
```

## 6. 修改后的执行流程

修改后的状态顺序应为：

```text
进入好友列表
→ 检查是否已经出现成功证据
    ├─ 是：点击小房子返回主界面
    └─ 否：继续
→ 第一次点击“领取赠送”
→ 等待2秒
→ 再次检查是否出现成功证据
    ├─ 是：点击小房子返回主界面
    └─ 否：继续
→ 第二次点击“领取赠送”
→ 再等待2秒
→ 再次检查是否出现成功证据
    ├─ 是：点击小房子返回主界面
    └─ 否：停止任务并保留现场
```

关键约束：

```text
未确认成功
→ 不允许点击小房子
```

## 8. OCR完成文本的处理要求

原代码只匹配：

```text
每日赠送上限30/30
```

但OCR结果可能包含：

```text
每日赠送上限：30/30
每日赠送上限:30/30
每日赠送上限 30/30
每日赠送上限：
30/30
```

因此必须先归一化文本，再判断：

```python
normalized_text = (
    page_text
    .replace(" ", "")
    .replace("\n", "")
    .replace("\r", "")
    .replace("：", "")
    .replace(":", "")
)
```

然后匹配：

```python
"每日赠送上限30/30" in normalized_text
```

不要只依赖完全一致的原始OCR字符串。

## 9. 不建议继续保留的旧逻辑

以下逻辑应删除：

```python
used_friendship_action = runner.has_decision(
    "友情点页面执行领取赠送"
)

if not used_friendship_action:
    # 点击领取赠送

returned_after_friendship = runner.has_decision(
    "友情点动作完成后回到主界面"
)

if not returned_after_friendship:
    # 直接返回主界面
```

主要问题是：

- `has_decision()` 只代表“生成过决策”；
- 没有任何成功确认；
- 点击和返回主页之间没有等待；
- 点击偶发失败后会永久跳过按钮；
- 再次进入好友页时可能继续读取旧历史；
- 容易形成跨轮次死循环。

## 10. 建议增加的调试信息

由于该问题属于低概率事件，请在不改变现有日志格式的前提下，尽量把下面的信息写入事件日志：

```python
{
    "task": "friendship",
    "page_text": page_text,
    "normalized_text": normalized_text,
    "friendship_success": friendship_success,
    "clicked_once": clicked_once,
    "waited_once": waited_once,
    "retried_once": retried_once,
    "waited_after_retry": waited_after_retry
}
```

如果事件记录器不支持直接写入这些字段，至少让执行日志输出：

```text
[friendship] success=False
[friendship] clicked_once=True
[friendship] waited_once=True
[friendship] retried_once=False
```

这样下次再出现低概率异常时，可以直接区分：

- OCR没有识别到成功文本；
- 首次点击没有生效；
- 历史决策提前命中；
- 重试没有执行；
- 等待阶段被跳过。

## 11. 进一步的推荐改进：任务局部状态

本次可以先采用上面的最小修改方案。

但从长期设计上看，不建议继续用全局历史决策表示任务状态。更可靠的方式是给友情点任务建立局部状态：

```python
{
    "clicked": False,
    "retry_count": 0,
    "confirmed": False
}
```

任务开始时重置，任务结束时清理。

示例思路：

```python
state = runner.task_state.setdefault(
    "friendship",
    {
        "clicked": False,
        "retry_count": 0,
        "confirmed": False
    }
)
```

但只有在项目已经存在 `task_state`、任务上下文或类似机制时才采用。不要为了本次修复盲目新增一套与 Runner 不兼容的状态系统。

当前优先级：

```text
第一优先：等待 + 成功确认 + 一次重试 + 失败停止
第二优先：任务局部状态重构
```

## 12. 测试用例

修改后至少测试以下场景。

### 场景A：正常领取成功

前提：

- 初始不是 `30/30`；
- 第一次点击正常生效；
- 页面出现“领取和赠送成功”。

预期：

```text
点击领取赠送
→ 等待
→ 识别成功
→ 返回主界面
```

不能重试第二次。

### 场景B：第一次点击偶发丢失

前提：

- 第一次点击没有生效；
- 等待后没有成功文本；
- 第二次点击正常生效。

预期：

```text
第一次点击
→ 等待
→ 未确认
→ 第二次点击
→ 等待
→ 确认成功
→ 返回主界面
```

不能在第一次点击后直接返回主页。

### 场景C：两次点击都失败

前提：

- 两次点击均未生效；
- 始终没有成功文本。

预期：

```text
第一次点击
→ 等待
→ 第二次点击
→ 等待
→ stop
```

必须停留在当前现场，不能点击小房子，不能重新从任务列表进入。

### 场景D：进入页面时已经达到30/30

前提：

OCR文本可能是：

```text
每日赠送上限：30/30
```

预期：

- 归一化后正确识别；
- 不点击“领取赠送”；
- 直接返回主界面。

### 场景E：OCR包含空格、换行和不同冒号

测试文本：

```text
每日赠送上限 30/30
每日赠送上限：30/30
每日赠送上限:
30/30
```

预期：全部能识别为完成状态。

### 场景F：旧决策历史仍然存在

前提：历史中已经存在旧的：

```text
友情点页面执行领取赠送
```

预期：新逻辑不再依赖这个旧intent，因为新代码使用了不同的阶段intent：

```text
友情点首次点击领取赠送
友情点首次点击后等待确认
友情点重试点击领取赠送
友情点重试后等待确认
```

如果 Runner 的历史在整个程序生命周期永久保留，仍需确认任务开始时是否会清理当前任务的决策记录。

## 13. 验收标准

- [ ] 未确认成功时不会点击小房子；

- [ ] 首次未成功时最多重试一次；

- [ ] 正常成功路径仍能返回主界面；

- [ ] 不改动其他任务逻辑；
- [ ] 日志中能够看出当前友情点任务处于哪个阶段；
- [ ] 不再出现“好友页—主页—任务列表—好友页”的无限往返。
